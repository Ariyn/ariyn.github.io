이번 마이그레이션의 핵심은 정적 사이트 생성기를 바꾸는 일이 아니라 운영 부담을 줄이는 일이었다.
Hugo가 부족해서가 아니라, 지금의 블로그에는 Jekyll과 GitHub Pages의 단순한 조합이 더 잘 맞았다.

{: .tractatus}
1. 이 블로그는 처음에 Hugo로 만들어졌다.

   1. 당시에는 Go를 자주 쓰고 있었다.

   2. 정적 사이트 생성기도 Go로 만들어진 Hugo가 자연스러운 선택이었다.

   3. 쓰고 싶던 테마도 Hugo 테마였기 때문에 선택 비용이 낮았다.

   4. 정적 파일을 만들어 github.io에 올리는 목적만 보면 Hugo는 충분히 좋은 도구였다.


2. Jekyll로 옮긴 이유는 GitHub Pages와의 궁합이었다.

   1. GitHub Pages는 Jekyll을 기본으로 지원한다.

   2. 저장소 구조만 맞으면 별도의 빌드 파이프라인을 강하게 유지하지 않아도 된다.

   3. 예전에는 익숙한 도구가 중요했지만, 지금은 계속 유지하기 쉬운 구조가 더 중요하다.

   4. 글쓰기 도구와 빌드 도구가 서로 부담을 만들면 결국 글을 덜 쓰게 된다.


3. 마이그레이션은 Codex를 사용해서 진행했다.

   1. 단순히 Markdown 파일만 옮긴 것이 아니라 테마와 레이아웃도 Jekyll에 맞게 정리했다.

   2. `_posts`의 front matter를 Jekyll 형식으로 맞췄다.

   3. 기존 slug와 permalink를 유지해서 가능한 한 예전 URL이 깨지지 않게 했다.

   4. 태그 페이지, 포스트 목록, RSS, sitemap 같은 주변 구조도 Jekyll 기준으로 다시 맞췄다.

   5. Hugo 테마의 HTML 조각은 Jekyll의 `_layouts`, `_includes`, `_plugins` 구조로 옮겼다.


4. 어려운 지점은 변환보다 보존 기준을 세우는 일이었다.

   1. 블로그 글은 코드와 다르게 사소한 변환 오류가 바로 드러나지 않는다.

   2. 본문은 최대한 그대로 두는 것이 맞았다.

   3. URL은 되도록 유지하는 것이 맞았다.

   4. 레이아웃과 빌드 구조만 Jekyll에 맞추는 것이 맞았다.

   5. 변환 스크립트는 다시 실행해도 결과가 크게 흔들리지 않아야 했다.


5. Codex는 반복 작업의 단위를 작게 만드는 데 유용했다.

   1. front matter 수정은 사람이 직접 하면 지루하고 실수하기 쉽다.

   2. 태그 페이지 생성과 permalink 확인도 반복성이 큰 작업이다.

   3. 기존 파일을 읽게 한 뒤 현재 블로그 규칙에 맞춰 변환시키면 빠르게 초안을 만들 수 있었다.

   4. 새 테마를 만드는 일과 기존 글을 깨지 않게 옮기는 일을 번갈아 처리할 수 있었던 점도 편했다.


6. 검증은 빌드 성공 여부만으로 충분하지 않았다.

   1. `jekyll build`가 성공해도 본문이 그대로 옮겨졌다는 뜻은 아니다.

   2. 원본 slug 목록과 Jekyll의 `/posts/{slug}/` 페이지 목록을 비교해야 했다.

   3. front matter를 제거한 본문끼리 hash를 비교해야 했다.

   4. 내부 링크와 이미지 경로가 유지되었는지도 따로 확인해야 했다.

   5. 렌더링된 article 본문의 텍스트, 링크 href, 이미지 src를 뽑아 의미 있는 차이만 확인하는 방식이 더 낫다.


7. 자동 변환은 편하지만 최종 판단을 대신하지는 않는다.

   1. URL이 유지되는지는 직접 확인해야 한다.

   2. Jekyll이 실제로 빌드되는지도 직접 확인해야 한다.

   3. Liquid 문법이 의도대로 동작하는지도 직접 확인해야 한다.

   4. 오래 남을 글과 링크를 다루는 작업에서는 자동화보다 검증 기준이 더 중요하다.


8. 결론은 Hugo가 아니라 운영 부담이 문제였다는 것이다.

   1. Hugo는 여전히 빠르고 좋은 도구다.

   2. 다만 지금의 목적은 복잡한 빌드 시스템을 유지하는 것이 아니다.

   3. 지금의 목적은 글을 쉽게 쓰고 오래 남기는 것이다.

   4. 정적 사이트 생성기는 글을 쓰기 위한 도구이지 그 자체가 목적이 아니다.
