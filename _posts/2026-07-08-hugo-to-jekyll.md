* 원래 이 블로그는 Hugo로 만들었다.
  * 당시에는 Go를 자주 쓰고 있었고, 정적 사이트 생성기도 Go로 만들어진 Hugo가 자연스러운 선택이었다.
  * 쓰고 싶던 블로그 테마도 Hugo 테마였기 때문에, 크게 고민할 이유가 없었다.
    * 정적 파일을 만들고 github.io에 올리는 목적만 보면 Hugo는 충분히 좋은 도구였다.
* 그런데 이번에는 블로그를 Jekyll로 옮겼다.
  * 가장 큰 이유는 github.io와의 궁합이다.
    * GitHub Pages는 Jekyll을 기본으로 지원한다.
    * 별도의 빌드 파이프라인을 강하게 유지하지 않아도, 저장소 구조만 맞으면 배포 흐름이 단순해진다.
  * 예전에는 "내가 익숙한 도구"가 중요했다면, 지금은 "블로그를 계속 유지하기 쉬운 구조"가 더 중요해졌다.
    * 글을 쓰는 도구와 블로그를 빌드하는 도구가 서로 부담을 만들면 결국 글을 덜 쓰게 된다.
* 마이그레이션은 Codex를 사용해서 진행했다.
  * 단순히 markdown 파일만 옮긴 것은 아니고, 테마와 레이아웃까지 Jekyll에 맞게 다시 정리했다.
    * `_posts`의 front matter를 Jekyll 형식으로 맞췄다.
    * 기존 slug와 permalink를 유지해서, 가능한 한 예전 URL이 깨지지 않도록 했다.
    * 태그 페이지, 포스트 목록, RSS, sitemap 같은 주변 구조도 Jekyll 기준으로 다시 맞췄다.
  * Hugo 테마에서 쓰던 HTML 조각은 Jekyll의 `_layouts`, `_includes`, `_plugins` 구조로 옮겼다.
    * 이 과정에서 템플릿 문법이 달라지는 부분이 있었다.
    * Hugo의 template 함수와 Jekyll의 Liquid는 비슷해 보여도, 실제로는 데이터 접근 방식과 필터 사용 방식이 다르다.
* 의외로 핵심은 변환 자체보다 "기존 글을 얼마나 건드리지 않을 것인가"였다.
  * 블로그 글은 코드와 다르게, 사소한 변환 때문에 문장이나 링크가 깨져도 바로 티가 나지 않는다.
  * 그래서 마이그레이션의 기준을 아래처럼 잡았다.
    * 글 본문은 최대한 그대로 둔다.
    * URL은 되도록 유지한다.
    * 레이아웃과 빌드 구조만 Jekyll에 맞춘다.
    * 변환 스크립트는 다시 실행해도 결과가 크게 흔들리지 않게 만든다.
* Codex를 사용해서 좋았던 점은 반복 작업의 단위가 작아졌다는 것이다.
  * front matter 수정, 태그 페이지 생성, permalink 확인 같은 작업은 사람이 직접 하면 귀찮고 실수하기 쉽다.
  * 하지만 Codex에게 기존 파일을 읽게 하고, 현재 블로그의 규칙에 맞춰 변환하도록 시키면 빠르게 초안을 만들 수 있었다.
  * 특히 "새 테마를 만드는 일"과 "기존 글을 깨지 않게 옮기는 일"을 번갈아 처리할 수 있는 점이 편했다.
* 그래서 검증은 빌드 성공 여부만 보지 않기로 했다.
  * `jekyll build`가 성공해도 본문이 그대로 옮겨졌다는 뜻은 아니다.
    * 정적 사이트 생성기는 HTML을 만들어줄 뿐이고, 마이그레이션 과정에서 문장, 링크, 이미지 경로가 바뀌었는지는 별도로 확인해야 한다.
  * 먼저 원본 글 목록과 변환된 글 목록을 비교했다.
    * Hugo에서 쓰던 slug 목록과 Jekyll에서 생성된 `/posts/{slug}/` 페이지 목록이 맞아야 한다.
    * 새로 작성한 글은 마이그레이션 대상이 아니므로, 기존 글 검증에서는 별도로 제외해야 한다.
  * 그 다음에는 front matter를 제거한 본문끼리 비교하는 쪽이 좋다.
    * `layout`, `permalink`, `tags` 같은 값은 Jekyll에 맞게 바뀔 수 있다.
    * 하지만 본문 markdown은 가능한 한 그대로여야 한다.
    * 따라서 앞뒤 공백과 줄바꿈 정도만 정규화한 뒤 hash를 비교하면 된다.
  * 마지막으로 렌더링된 결과를 비교한다.
    * 테마가 바뀌면 HTML 전체는 당연히 달라진다.
    * 대신 article 본문 안의 텍스트, 링크 href, 이미지 src만 뽑아서 비교하면 의미 있는 차이만 볼 수 있다.
  * 결국 검증 기준은 아래와 같이 잡는 것이 좋았다.
    * 원본 slug 목록과 변환된 slug 목록이 같은가?
    * 원본 본문과 변환된 본문의 hash가 같은가?
    * 내부 링크와 이미지 경로가 유지되었는가?
    * Jekyll 빌드가 성공하는가?
* 물론 Codex가 알아서 다 해주는 것은 아니다.
  * URL이 유지되는지, Jekyll이 실제로 빌드되는지, Liquid 문법이 의도대로 동작하는지는 직접 확인해야 한다.
  * 자동 변환은 편하지만, 블로그처럼 오래 남을 글과 링크를 다루는 작업에서는 검증이 더 중요하다.
* 정리하면, 이번 마이그레이션은 Hugo가 싫어서가 아니라 운영 부담을 줄이기 위한 선택이었다.
  * Hugo는 여전히 빠르고 좋은 도구다.
  * 다만 내 블로그의 현재 목적은 복잡한 빌드 시스템을 유지하는 것이 아니라, 글을 쉽게 쓰고 오래 남기는 것이다.
  * 블로그를 더 자주 고치고, 더 쉽게 배포하고, 덜 신경 쓰려면 Jekyll과 GitHub Pages의 단순한 조합이 더 잘 맞았다.
    * 정적 사이트 생성기는 글을 쓰기 위한 도구이지, 그 자체가 목적이 아니니까.
