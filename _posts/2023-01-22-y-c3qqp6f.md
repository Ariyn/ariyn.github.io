정체도 알 수 없는 패키지를 임포트하고 싶지 않아 JwtTokenFilter를 직접 구현했다
`org.springframework.web.filter.OncePerRequestFilter`를 상속받아, `doFilterInternal`을 구현하고, 인증이 되었다면 `SecurityContextHolder`에 authentication을 추가해주면 되는 간단한 작업이었다.

{: .tractatus}
1. 정체도 알 수 없는 패키지를 임포트하고 싶지 않아 JwtTokenFilter를 직접 구현했다

   1. `org.springframework.web.filter.OncePerRequestFilter`를 상속받아, `doFilterInternal`을 구현하고, 인증이 되었다면 `SecurityContextHolder`에 authentication을 추가해주면 되는 간단한 작업이었다.

      1. authentication은 `org.springframework.security.authentication.UsernamePasswordAuthenticationToken`을 사용했다

         ```java
                 UsernamePasswordAuthenticationToken(userDetails, null)
         ```

      2. [출처](https://www.toptal.com/spring/spring-security-tutorial)

   2. 그러나 403 forbidden만 뜨면서 진행이 안된다

   3. 수많은 페이지를 찾아보고, 오랜시간 step into를 눌러가면서 확인한 결과, authentication이 granted false 되는것을 확인했다.

      ```java
            org/springframework/security/web/access/intercept/AuthorizationFilter.java:95
            try {
              AuthorizationDecision decision = this.authorizationManager.check(this::getAuthentication, request);
              ...

            ---
              inspector
            decision = AuthorizationDecision [granted=false]
      ```

   4. 원인을 찾아보니, UsernamePasswordAuthenticationToken의 생성자에 숨어 있었다.

      ```java
            public UsernamePasswordAuthenticationToken(Object principal, Object credentials) {
                super(null);
                this.principal = principal;
                this.credentials = credentials;
                setAuthenticated(false);
            }

            public UsernamePasswordAuthenticationToken(Object principal, Object credentials,
                        Collection<? extends GrantedAuthority> authorities) {
                super(authorities);
                this.principal = principal;
                this.credentials = credentials;
                super.setAuthenticated(true); // must use super, as we override
            }

            public static UsernamePasswordAuthenticationToken authenticated(Object principal, Object credentials,
                    Collection<? extends GrantedAuthority> authorities) {
                return new UsernamePasswordAuthenticationToken(principal, credentials, authorities);
            }
      ```

      1. 마지막의 authorities가 없이 호출한다면 `isAuthenticated = false`로 설정되어서 진행이 안된 것이었다.

      2. 참고로 외부에서 함부로 setAuthenticated를 호출한다면 아래와 같은 로그 메시지를 받는다

         ```java
                 Cannot set this token to trusted - use constructor which takes a GrantedAuthority list instead

                 org/springframework/security/authentication/UsernamePasswordAuthenticationToken.java:112
                 @Override
                 public void setAuthenticated(boolean isAuthenticated) throws IllegalArgumentException {
                     Assert.isTrue(!isAuthenticated,
                             "Cannot set this token to trusted - use constructor which takes a GrantedAuthority list instead");
                     super.setAuthenticated(false);
                 }
         ```

   5. 정리하면 `UsernamePasswordAuthenticationToken`은 생성자에 authorities를 넘겨야 authenticated 상태가 된다.

   6. `UsernamePasswordAuthenticationToken(userDetails, null)`처럼 principal과 credentials만 넘기면 인증 전 token으로 만들어진다.

   7. 인증이 끝난 사용자라면 authorities를 포함하는 생성자나 `authenticated(...)` factory method를 사용해야 한다.

   8. 이 지점이 헷갈리는 이유는 클래스 이름만 보면 token을 만든 순간 인증된 것처럼 느껴지기 때문이다.

   9. 실제로는 같은 타입이 인증 전/후 상태를 모두 표현한다.

   10. 그래서 filter에서 직접 `Authentication`을 만들 때는 `isAuthenticated()` 값과 authorities를 반드시 확인해야 한다.

   11. 직접 JWT filter를 구현할 때의 체크포인트는 단순하다.

   12. token 검증 후 사용자 정보를 만든다.

   13. 사용자 권한 목록을 만든다.

   14. authenticated 상태의 `UsernamePasswordAuthenticationToken`을 만들고 `SecurityContextHolder`에 넣는다.

   15. 403이 나오면 `AuthorizationFilter`에서 decision이 왜 `granted=false`가 되었는지 확인한다.
